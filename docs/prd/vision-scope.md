# DeepNote AI: Obsidian-Class Knowledge Management Upgrade
## Vision & Scope Document

**Version:** 1.0
**Date:** 2026-04-06
**Status:** Draft

---

## Research Foundation

This upgrade is informed by deep analysis of two converging paradigms in knowledge management:

**Obsidian as the benchmark.** Obsidian has become the de facto standard for local-first knowledge management, with 1800+ community plugins, a markdown-native editing experience, bidirectional linking, graph visualization, and a canvas for spatial thinking. Its strength is composability: users assemble their own workflows from primitives. Its weakness is that AI integration is bolted on through third-party plugins with no unified intelligence layer.

**Karpathy's LLM Wiki concept.** Andrej Karpathy articulated a gap in current AI tools: chat-based RAG is stateless. You ask a question, get an answer, and the system forgets. An LLM Wiki inverts this -- the AI continuously maintains structured, persistent wiki pages synthesized from ingested sources. Instead of retrieving raw chunks and hoping the LLM stitches them together, the system maintains a living knowledge base that the AI has already reasoned over. Coverage indicators show what the AI knows well vs. where gaps exist. Queries hit the wiki first, falling back to RAG only when the wiki lacks coverage.

**The opportunity.** DeepNote already has both pieces -- AI-powered source ingestion with RAG, and a notes system for human knowledge capture. But these exist in separate silos. The notes system is a plain textarea with no structure. The AI has no persistent memory beyond conversation context. This upgrade fuses Obsidian's knowledge management UX with an AI-native intelligence layer that Obsidian cannot replicate without a fragile stack of plugins, creating a tool where human notes and AI-synthesized knowledge live in the same graph, link to the same sources, and reinforce each other.

**The narrowest wedge.** The entire upgrade begins with one change: replacing the plain textarea note editor with a Tiptap-based markdown editor that renders live preview. This single change unlocks every subsequent tier -- you cannot have block references without blocks, cannot have a knowledge graph without parseable links, cannot have wiki pages without rich formatting. The editor is the foundation.

## Product Vision

Transform DeepNote AI from a research assistant with a scratchpad into a research assistant with an Obsidian-class knowledge management system where human notes and AI-synthesized wiki pages coexist in a single, interconnected knowledge graph.

## Problem Statement

DeepNote's current notes system has five specific limitations that prevent it from serving as a serious knowledge management tool:

1. **Plain textarea editing.** Notes are unformatted text. There is no markdown rendering, no headings hierarchy, no code blocks, no math, no callouts, no embeds. Users who want formatted notes must preview them mentally. This is the single biggest barrier to adoption as a primary notes tool.

2. **Flat organization.** All notes exist in a single list. There are no folders, no nested hierarchy, no quick switcher, no full-text search across notes. Users with more than a few dozen notes cannot find anything efficiently.

3. **Disconnected links.** While `[[wiki links]]` and `#tags` are supported syntactically, there is no graph view, no backlink panel, no unlinked mention detection, and no way to visualize how notes relate. The linking syntax exists but the knowledge graph does not.

4. **No workflow primitives.** There are no templates, no daily notes, no periodic notes, no task management, no dataview-style queries. Users cannot build repeatable research workflows within the notes system.

5. **AI and notes are siloed.** The AI assistant ingests sources and answers questions, but it does not maintain persistent knowledge. Notes written by the user and insights generated by the AI exist in completely separate systems. There is no AI-maintained wiki, no automatic linking of AI outputs to relevant notes, no coverage map showing what topics the AI understands well. The human knowledge graph and the AI knowledge base never intersect.

## Anti-Goals (What This Is NOT)

1. **Not a standalone notes app.** This is not a competitor to Obsidian, Notion, or Logseq as a general-purpose notes tool. Every feature decision is evaluated in the context of AI-assisted research. Features that only make sense for generic note-taking (e.g., habit trackers, recipe managers) are out of scope.

2. **Not a plugin ecosystem.** We will not build a plugin API, marketplace, or community extension system. Obsidian's plugin model is its core competitive advantage and attempting to replicate it is a multi-year distraction. We build the features we need directly into the app.

3. **Not a cloud sync service.** DeepNote is local-first. We will not build cloud storage, real-time collaboration, or sync infrastructure. If sync is needed in the future, it will be through standard file system tools (Git, Syncthing, iCloud Drive) rather than a proprietary service.

4. **Not a replacement for the existing AI chat.** The knowledge wiki and AI-native features augment the existing chat and RAG pipeline. Chat remains the primary interaction mode for asking questions. The wiki is a persistent layer that makes chat answers better over time, not a replacement for conversational AI.

5. **Not a full Obsidian feature-parity effort.** We are building the 20% of Obsidian's features that deliver 80% of the value for research workflows. We will not implement every Obsidian plugin's functionality. Specifically, we are not building: publish/hosting, mobile-specific UX, Vim mode, or cosmetic theming beyond light/dark.

6. **Not a separate product.** This is an upgrade to the existing DeepNote app. The notes system, the AI chat, the source ingestion, and the Studio all remain part of a single integrated application. There will be no separate "DeepNote Notes" product or pricing tier.

7. **Not AI-generated-only content.** The knowledge wiki is AI-maintained but human-governed. Users can edit, override, pin, or delete any wiki page. The AI proposes; the human disposes. We will not build a system where the AI autonomously publishes content without user visibility and control.

## Primary User

**The research-focused knowledge worker** who already uses DeepNote AI for source ingestion and AI-assisted questioning, and is frustrated that their research notes live in a separate tool (Obsidian, Notion, or plain text files).

This user:
- Ingests 10-50+ sources per research project (papers, articles, documents)
- Takes extensive notes while reading and during AI conversations
- Wants to see connections between ideas across sources and their own thinking
- Values local-first data ownership and does not want their research in someone else's cloud
- Is comfortable with markdown syntax and expects it to render properly
- Currently uses a second app for knowledge management because DeepNote's notes are too basic
- Would consolidate into a single tool if the notes system were powerful enough

They do not need DeepNote to replace their task manager, calendar, or project management tool. They need it to be the single place where research sources, AI-generated insights, and personal notes converge into a navigable knowledge graph.

## Success Criteria

1. **Editor adoption.** Within 30 days of the editor upgrade shipping (Tier 1), 80% of active users create at least one note using markdown formatting features (headings, bold, code blocks, or links). Measured via anonymized local telemetry opt-in or manual user survey.

2. **Notes volume increase.** Average notes per user increases by 3x within 60 days of Tiers 1-2 shipping, compared to the 60-day period before the upgrade. A capable editor and proper organization remove the friction that currently discourages note-taking.

3. **Single-tool consolidation.** In user interviews or surveys, at least 50% of users who previously used a separate notes tool (Obsidian, Notion, etc.) report that DeepNote is now their primary research notes tool within 90 days of Tiers 1-3 shipping.

4. **Graph engagement.** After Tier 3 ships, at least 40% of weekly active users open the graph view at least once per week. Graph views that are opened but immediately closed do not count -- the user must interact (click a node, filter, or navigate) for it to register as engagement.

5. **Wiki coverage utilization.** After Tier 6 (Knowledge Wiki) ships, at least 60% of users with 5+ ingested sources have AI-generated wiki pages covering their source material. Of those, at least 30% have edited or annotated at least one wiki page, indicating they find the AI synthesis valuable enough to engage with directly.

6. **Query-first hit rate.** After Tier 6 ships, at least 40% of AI chat questions on topics covered by wiki pages are answered from the wiki layer without falling back to raw RAG chunk retrieval. This measures whether the wiki is actually improving answer quality and reducing retrieval latency.

7. **Retained engagement.** 30-day retention (defined as opening DeepNote and interacting with either notes or chat on at least 8 of 30 days) increases by 25% compared to pre-upgrade baseline. A knowledge management system that users build their thinking in creates daily-use stickiness that a chat-only tool does not.

## Tiered Delivery Roadmap (Summary)

Each tier is shippable independently and provides standalone value. Later tiers build on earlier ones but are not blocked by tiers that follow them.

| Tier | Name | Core Deliverable | Depends On |
|------|------|-----------------|------------|
| 1 | Editor Upgrade | Tiptap markdown editor with live preview, callouts, math, Mermaid, embeds, block references | Nothing (foundation) |
| 2 | Organization | Folder tree, quick switcher, command palette, full-text search, note outline, bookmarks, frontmatter properties | Tier 1 |
| 3 | Knowledge Graph | Interactive graph view (global + local), backlink panel, unlinked mention detection, graph filters | Tier 1 |
| 4 | Workflows | Daily notes, templates, periodic notes, task checkboxes, kanban view, dataview-style queries | Tiers 1-2 |
| 5 | Canvas | Infinite spatial canvas with embedded notes, freeform cards, directional arrows, grouping | Tiers 1-2 |
| 6 | Knowledge Wiki | AI-maintained wiki pages from sources, coverage indicators, wiki lint, query-first-then-RAG pipeline | Tiers 1-3 |
| 7 | AI-Native Enhancements | Auto-tagging, auto-linking, smart semantic search, note summarization, content generation from notes, voice-to-note, web clipper | Tiers 1-3, 6 |

Tiers 3 and 4 can be developed in parallel. Tier 5 can begin after Tier 2. Tier 7 can begin partially after Tier 3 but requires Tier 6 for full capability.

## Key Technical Constraints

- **Tiptap is already in the codebase** (used in Studio). The editor upgrade extends its usage to notes, not introduces a new dependency.
- **SQLite + Drizzle ORM** is the persistence layer. All new data (folder hierarchy, graph edges, wiki pages, frontmatter properties) must fit within SQLite's capabilities. No external database dependencies.
- **Electron main/renderer split** means all file I/O and heavy computation (graph layout, wiki generation, embedding) happens in the main process or a worker, never in the renderer.
- **Multi-provider AI** means wiki generation and AI-native features must work with any configured provider (Gemini, Claude, OpenAI, Groq), not be hardcoded to one.
- **Existing IPC contract** (`IpcHandlerMap` in `src/shared/types/ipc.ts`) must be extended for new features following established patterns.

## Risk Summary

| Risk | Impact | Mitigation |
|------|--------|------------|
| Editor migration breaks existing notes | High | Tier 1 includes a migration path that preserves all existing note content. Plain text remains valid markdown. |
| Knowledge graph performance at scale | Medium | Lazy loading, WebGL rendering (e.g., force-graph), and graph pruning for notebooks with 1000+ notes. |
| AI wiki quality varies by provider | Medium | Provider-agnostic prompts with quality scoring. Wiki pages include confidence indicators. Users can regenerate with a different provider. |
| Scope creep toward "build all of Obsidian" | High | Anti-goals document is the guardrail. Every feature must pass the test: "Does this serve AI-assisted research, or is this generic note-taking?" |
| Tiptap performance with large documents | Medium | Virtual rendering for documents over 10,000 words. Benchmark with realistic research note sizes during Tier 1 development. |
